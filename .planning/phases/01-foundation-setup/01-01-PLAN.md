4. Write `package.json` at plugin root with:
   - "name": "vault-plugin-deps"
   - "version": "1.0.0"
   - "private": true
   - "description": "Dependencies for vault plugin scripts (installed to CLAUDE_PLUGIN_DATA)"
   - "dependencies": { "gray-matter": "^4.0.3", "fast-glob": "^3.3.0", "js-yaml": "^4.1.0" }

5. Write `.gitattributes` with two lines: `scripts/*.sh eol=lf` and `scripts/*.sh diff`

6. Create directory `agents/` and write an empty `agents/.gitkeep` file.

**A. skills/vault-init/SKILL.md** -- Full implementation skill (per D-01, D-02, D-03). This is the ONLY skill with a full body in Phase 1.

Frontmatter fields:
- name: vault-init
- description: Initialize or adapt an Obsidian vault with PARA structure, templates, and agent context files
- disable-model-invocation: true
- allowed-tools: Read Write Bash Glob Grep
- argument-hint: "[--dry-run]"

Body content must be the complete init wizard instructions. The body tells Claude exactly what to do when the user runs /vault:init. It MUST include these sections in this order:

Section "# /vault:init" -- Title and one-line description.

Section "## How This Works" -- Explain: you are running inside the user's Obsidian vault directory (CWD per D-02). Detect state, build proposal, present, execute only after confirm.

Section "## Step 1: Detect Vault State" -- Scan CWD for vault markers: .obsidian/ directory, existing PARA folders (both numbered like "00 Inbox/" and unnumbered like "inbox/"), organization.md or vault:config.md, any .md files. Classify as Greenfield (no .obsidian/ and no .md files or only README.md) or Brownfield (has .obsidian/ or has existing .md content). Report findings.

Section "## Step 2: Build Proposal" -- For greenfield: propose full PARA structure (00 Inbox/, 01 Projects/, 02 Areas/, 03 Resources/, 04 Archive/, 10 MOCs/, 20 Templates/ with Project Dashboard.md + Task List.md + Inbox Note.md, 30 Attachments/, 40 Daily Notes/) plus vault:config.md, CLAUDE.md pointer, AGENTS.md pointer, Home.md. For brownfield: identify existing vs missing PARA folders, existing templates, check for organization.md (propose merge into vault:config.md per D-04), check for existing CLAUDE.md (update not replace), suggest per-file migrations with explicit approval required (per D-03). Build summary: folders to CREATE, folders that EXIST, templates to INSTALL (never overwrite), config to GENERATE, migration suggestions.

Section "## Step 3: Present Proposal" -- Show complete proposal. Separate "will create" from "suggests moving". Ask: "Approve this plan? (yes / yes with changes / no)"

Section "## Step 4: Execute" -- Only after approval. Steps: (1) Create missing PARA folders with mkdir -p, (2) Install templates from ${CLAUDE_SKILL_DIR}/templates/ -- read bundled template, write to vault's 20 Templates/ folder, never overwrite existing, (3) Generate vault:config.md with four sections per D-04: Folder Conventions, Frontmatter Spec, Linking Rules, Agent Instructions, with generated: timestamp in frontmatter, (4) Generate/update CLAUDE.md with vault context pointer to vault:config.md per D-05, (5) Generate AGENTS.md with pointer to vault:config.md per D-05, (6) Execute only user-approved brownfield migrations.

Section "## Step 5: Verify" -- Report folders created, templates installed, config files generated. Verify vault:config.md contains all four sections. Confirm CLAUDE.md and AGENTS.md reference vault:config.md.

Section "## --dry-run Flag" -- If $ARGUMENTS contains --dry-run, execute only Steps 1-3. Do NOT execute Step 4.

Section "## Important Rules" -- NEVER move or rename a file without explicit user approval for THAT specific file. NEVER overwrite an existing template. ALWAYS use numbered PARA prefixes (00, 01, 02, 03, 04, 10, 20, 30, 40). ALWAYS include generated: YYYY-MM-DD in vault:config.md frontmatter. vault:config.md is the CANONICAL conventions source.
